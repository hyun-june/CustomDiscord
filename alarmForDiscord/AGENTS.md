# 저장소 가이드라인

## 프로젝트 구조 및 모듈 구성

이 저장소는 현재 `alarmForDiscord`를 위한 빈 스캐폴드 상태입니다. 미래의 기여자들이 기능을 빠르게 찾을 수 있도록 예측 가능한 구조로 프로젝트 파일을 추가하세요:

- `src/`: 애플리케이션 소스 코드.
- `tests/`: `src/` 경로를 미러링하는 자동화 테스트.
- `assets/`: 아이콘, 소리 또는 샘플 데이터와 같은 정적 파일.
- `docs/`: 상세 설계 노트, 설정 참조 및 운영 런북.
- `.env.example`: 비밀 정보가 제외된 문서화된 설정 키 예시.

생성된 출력물, 의존성 폴더 및 로컬 런타임 파일은 버전 관리에서 제외하세요.

## 빌드, 테스트 및 개발 명령

빌드 시스템이나 패키지 매니페스트가 아직 존재하지 않습니다. 추가될 때 여기에 표준 명령을 문서화하고 일반적인 스크립트 이름을 사용하세요. 예시:

- `npm install`: 프로젝트가 Node.js를 사용하는 경우 JavaScript 의존성을 설치합니다.
- `npm run dev`: 로컬 개발 프로세스를 시작합니다.
- `npm test`: 자동화된 테스트 스위트를 실행합니다.
- `npm run build`: 프로덕션용 결과물을 생성합니다.

다른 스택이 선택되면 이 예시들을 해당 저장소의 실제 명령으로 교체하세요.

## 코딩 스타일 및 명명 규칙

도구가 구성되면 프로젝트 도구에서 강제하는 스타일을 사용하세요. 그 전까지는 코드를 단순하고 일관되게 유지하세요:

- JavaScript, TypeScript, JSON, YAML 및 Markdown에 대해 2칸 들여쓰기를 사용하세요.
- `discordClient.ts`, `alarmScheduler.ts`, `configLoader.ts`와 같이 서술적인 파일 이름을 사용하세요.
- 변수와 함수에는 `camelCase`, 클래스와 컴포넌트에는 `PascalCase`, 환경 상수는 `UPPER_SNAKE_CASE`를 권장합니다.
- 모듈은 하나의 책임에 집중하도록 유지하세요.

코드베이스가 커지기 전에 포맷터 및 린터 스크립트를 추가하세요.

## 테스트 가이드라인

아직 테스트 프레임워크가 구성되지 않았습니다. 테스트는 `tests/` 아래에 두거나 `.test.ts`, `.spec.ts` 또는 선택한 언어의 관례에 따라 소스 파일 옆에 배치하세요.

스케줄링 로직, Discord API 통합 경계, 설정 파싱 및 오류 처리에 대한 테스트를 우선시하세요. 일반적인 테스트에서는 실제 Discord 메시지를 보내는 대신 외부 서비스를 모킹(Mock)하세요.

## 커밋 및 풀 리퀘스트 가이드라인

이 디렉토리는 현재 Git 저장소가 아니므로 로컬 커밋 규칙을 유추할 수 없습니다. 다음과 같이 간결하고 명령조인 커밋 메시지를 사용하세요:

- `Add alarm scheduling service`
- `Validate Discord webhook configuration`

풀 리퀘스트에는 짧은 요약, 테스트 결과, 해당하는 경우 연결된 이슈, 사용자에게 보이는 동작에 대한 스크린샷 또는 로그를 포함해야 합니다. 설정 변경 사항 및 필요한 마이그레이션 단계를 명시하세요.

## 보안 및 설정 팁

Discord 토큰, 웹훅 URL 또는 사용자 고유의 비밀 정보를 절대로 커밋하지 마세요. 로컬 값은 `.env`에 저장하고 `.env.example`만 커밋하세요. 시작 시 필수 설정을 검증하고 민감한 값을 로그에 남기지 마세요.
